# OpenGameMate Phase 0 可行性报告

报告日期：2026-07-14

## 结论

当前结论为 **Conditional Go：Phase 0 的八项验证均已获得至少一次代码、运行或用户人工证据**。

解决方案结构、WebView2 独立用户数据目录、人工登录与网页麦克风、主显示器截图、最小证据日志和隐私边界已经得到编译、运行或用户实测证据。最高优先级的“后台无焦点加入图片和文字”已通过 `composer-paste` 路径完成一次真实验证；用户明确确认后的后台提交也已触发，输入框和附件清空且前台焦点、鼠标未变化。

“Conditional”表示关键网页自动化只完成单次成功运行，当前还处于任意 HTTPS 导航诊断模式，且 ChatGPT DOM、事件机制、账号能力或页面版本变化都可能导致回归。可以据此进入下一阶段规划，但在恢复正式导航边界和完成重复稳定性测试前，不应宣称完整产品已可发布。

## 实验中发现的问题

### WebView2 宿主句柄初始化顺序

首版在浏览器窗口尚未显示、WPF HWND 尚未建立时调用 `EnsureCoreWebView2Async`，初始化会一直等待且没有第二个窗口。修复为先显示浏览器宿主窗口，再初始化 WebView2。修复后实际出现独立 ChatGPT 窗口。

### 桌面启动方式差异

桌面控制接口直接用 exe 路径启动时没有创建进程，也没有返回错误；通过资源管理器打开同一 exe 后正常运行。这是测试控制接口的限制，不是 PoC 自身的启动失败。正常用户可直接双击 exe 或从 PowerShell 启动。

### ChatGPT 首次访问的人机验证

独立 UDF 意味着没有继承现有浏览器会话。首次打开 `chatgpt.com` 出现 Cloudflare“验证您是真人”，因此无法在无人参与的环境中继续登录和真实页面实验。这是预期的外部验证边界，不能绕过或伪造完成。

### 登录弹窗在旧 WebView2 131 中持续加载

本机系统 WebView2 Runtime 停留在 `131.0.2903.63`（2024-11），`edgeupdate` 服务为 Disabled，且没有组策略要求禁用更新。登录弹窗在该运行时中只能显示旋转加载图标。微软 Evergreen Bootstrapper 因机器级 Runtime 已存在而返回 `0x80040828`，恢复更新服务又因当前会话没有管理员权限被 Windows 拒绝。

PoC 因此增加仓库外隔离 Fixed Runtime 支持。安装 Microsoft 签名有效的 `150.0.4078.65` x64 Runtime 后，PoC 日志确认实际使用 150；同一登录弹窗在 5 秒内完整显示 Google、Apple、手机号、邮箱和可用的继续按钮。该故障判定为旧系统 WebView2 兼容性问题，已在 PoC 范围内修复。

### 邮箱继续后跳转微软身份端点

用户在登录弹窗输入邮箱并点击“继续”后，ChatGPT 请求导航到 `login.microsoftonline.com`。旧守卫只允许 OpenAI/ChatGPT 域名，因而取消请求并造成持续加载。该主机与 Microsoft 官方 OAuth 授权端点一致，OpenAI SSO 文档也说明登录可能转交身份提供商。

经用户明确授权，PoC 只允许精确 HTTPS 主机 `login.microsoftonline.com` 和默认端口；首次跳转弹窗确认且授权只在当前运行有效。HTTP、任意子域名、伪造后缀、非默认端口和其他外部域名仍被阻断。程序不记录 URL 路径、查询参数、账号、凭据、Cookie 或令牌。代码已构建通过且测试 21/21；最终登录仍须用户本人验证，不能仅凭测试判定通过。

人工复测发现该例外仍不足：微软第一跳获准后继续重定向到 `login.live.com`，Google 登录则导航到 `accounts.google.com`；两者均被当前守卫取消并产生 `OperationCanceled`。因此“弹窗确认后无跳转”和“Google 持续加载”已判定为外部身份链白名单过窄，而不是 Runtime 或登录按钮故障。在用户进一步授权前，这两个来源继续保持阻断。

为优先完成 Phase 0 可行性验证，用户随后明确要求暂不实施域名限制。当前 PoC 已切换为临时 HTTPS 导航诊断模式：任意 HTTPS 主机可作为顶层导航或新窗口目标，无效 URI 和非 HTTPS 导航仍被阻断。该模式只放宽导航主机，不放宽数据读取、自动登录、权限或消息发送边界；日志仍只记录主机与协议。诊断版已构建通过并通过 24/24 测试，真实 ChatGPT 首页已在 Fixed Runtime 150 中重新加载，最终登录结果待用户手动验证。

### 第 5 项首次进入 DOM 阶段时的 PoC 解析错误

用户人工登录后，前两次实验因倒计时结束时 OpenGameMate 仍在前台而安全中止。第三次正确切换到其他程序后进入 DOM 阶段，但 PoC 抛出 `KeyNotFoundException`，图片和文字均未加入。根因是适配器把标准 `Runtime.evaluate` 返回结构 `result.value` 错误解析成 `result.result.value`。该错误发生在实际页面修改之前，不能判定为 ChatGPT 不支持后台输入。

解析器已改为显式验证并读取顶层 `result.value`；页面异常或返回结构缺失时失败关闭，不再暴露字典键异常。修复后构建 0 警告、0 错误，测试 27/27。真实第 5 项仍需用户重新执行，当前状态保持“待验证”。ChatGPT 窗口在实验开始时隐藏、结束后不自动显示是有意设计，用于避免抢焦点污染证据。

### 当前页面没有持久文件输入控件

用户使用解析器修复版再次执行两次实验，窗口与鼠标稳定检查均通过，但结果为 `fileSelected=False`、`previewDetected=False`、`textInserted=False`、`code=file-input-count`。这说明本轮已经越过解析器，失败点是旧适配器要求页面初始 DOM 恰好存在一个图片文件输入控件；当前 ChatGPT 页面不满足该假设。该结果仍不是“后台输入能力 No-Go”，因为附件事件尚未真正送达输入区。

PoC 现保留原 CDP 文件输入路径，并增加窄范围回退：若无法唯一定位文件输入，则只向唯一的 `#prompt-textarea` 派发包含固定测试 PNG 的粘贴事件，不打开原生文件选择器。自动判定要求图片文件已进入文件控件，或页面在输入区范围内出现附件预览；同时固定文字必须一致，前台窗口与鼠标必须稳定。Release 构建 0 警告、0 错误，测试 30/30；真实页面是否接受合成粘贴事件仍待用户复测，不能因构建通过宣称成功。

用户随后在真实已登录页面复测成功。截图中图片预览和固定测试文字同时可见；最小日志记录 `previewDetected=True`、`textInserted=True`、`focusAndCursorStable=True`，且附件采用 `composer-paste` 路径。第 5 项因此记为“单次真实运行通过”。该结论不包含第 6 项真实提交，也不代表页面更新后的长期稳定性。

用户人工确认附件预览后执行一次真实提交。最小日志记录 `triggerInvoked=True`、`composerCleared=True`、`attachmentCleared=True`、`focusAndCursorStable=True`、`code=ok`，截图中可见固定测试图片和文字已作为用户消息进入对话。第 6 项因此记为“单次真实运行通过”。测试文字明确要求 “Please do not respond”，所以未出现助手回复与本项结果不矛盾；同时按照隐私边界，PoC 不读取或使用回复正文作为证据。

### Voice 结束与 WebView2 关闭生命周期

用户人工确认网页 Voice 麦克风工作正常，但发现关闭 ChatGPT 子窗口后语音仍在运行。根因位于 PoC 而不是麦克风授权：`BrowserWindow` 原先取消 WPF `Closing` 事件并调用 `Hide()`，因此右上角关闭只是隐藏窗口，WebView2 与麦克风会话没有销毁。

OpenAI 官方 Voice 说明将麦克风控件定义为静音/取消静音，将退出控件定义为结束语音。PoC 现明确区分三种动作：隐藏只用于后台实验且不结束 Voice；正常情况由用户使用网页退出控件；网页控件无效时，可点击主界面“关闭 ChatGPT（结束语音）”或关闭子窗口，真正 Dispose WebView2。独立 UDF 保留，之后可以重新初始化而不主动删除登录状态。修复后的 Release 构建 0 警告、0 错误，测试 30/30；实际释放麦克风仍需用户在新版中复测。

## 第 5 项失败时的可接受替代方案

如果用户完成登录后，第 5 项仍因 DOM、可信用户手势、后台节流或焦点限制失败，Phase 0 应给出 No-Go，不继续构建完整自动化应用。可接受的降级方向按推荐顺序为：

1. **前台用户确认模式**：PoC 捕获并准备图片/固定文字，显示 WebView2，由用户本人点击上传/粘贴和发送。最可靠，但不满足“游戏不中断、后台无焦点提交”。
2. **仅准备、不提交**：后台生成截图和文本，向用户显示通知；用户方便时切到 WebView2 完成添加与发送。保留隐私边界，降低自动化承诺。
3. **官方受支持接口模式**：如果未来产品范围允许，改用有明确授权、稳定契约的官方 API，而不是适配 ChatGPT 消费者网页 DOM。该方向属于后续重新立项，不应在本 Phase 0 自动扩展。

不接受模拟鼠标、抢占游戏焦点、读取登录 Cookie/令牌或绕过人机验证作为替代方案。

## 下一阶段建议

当前只建议完成关闭生命周期和稳定性复测：

1. 在新版 PoC 中开始一次 Voice，分别验证网页退出控件和 PoC 的兜底关闭按钮；确认麦克风占用被释放后重新初始化，检查登录状态仍可用；
2. 在不继续真实发送的前提下重复若干次后台加入实验，观察页面版本和运行时稳定性；
3. Phase 1 规划时恢复正式导航边界，并把当前 DOM/粘贴路径视为需要回归测试的页面适配层。

Phase 0 当前为 Conditional Go，可以开始 Phase 1 规划，但不应在本阶段继续实现完整应用；若重复实验不稳定，则采用上述降级方案。
